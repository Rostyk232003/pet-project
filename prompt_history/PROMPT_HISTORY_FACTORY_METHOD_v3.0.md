## PROMPT v3.0: Гнучка система знижок через Factory Method

### Опис фічі
Впровадити гнучку систему нарахування знижок для замовлень, використовуючи патерн Фабричний метод. Основна ідея — відокремити процес вибору типу знижки від математичного розрахунку та логіки застосування в контролері.

### Архітектурні компоненти
1. **IDiscountProduct** — інтерфейс продукту (метод Apply(decimal price)).
2. **Concrete Products** — FirstOrderDiscount, HolidayDiscount, DefaultNoDiscount (чиста математика).
3. **DiscountCreator** — абстрактний творець (CreateDiscount, CalculateFinalPrice).
4. **Concrete Creators** — FirstOrderCreator, HolidayCreator, DefaultCreator (кожен повертає свій продукт).

---

### Варіанти вирішення (ToT)

**Варіант 1. Класичний Factory Method (1:1)**
- Для кожного типу знижки — окремий Creator та Product.
- Контролер працює лише з DiscountCreator, не знає про продукти.
- Всі розрахунки інкапсульовані у Product.
- Плюси: максимальна ізоляція, прозорість, легке розширення.
- Мінуси: при великій кількості знижок — багато класів.

**Варіант 2. Factory Method + Strategy**
- Один Creator, CreateDiscount приймає параметр типу знижки.
- Вибір продукту делегується через switch/словник.
- Плюси: менше класів, простіше DI.
- Мінуси: порушення SRP, Creator знає про всі продукти.

**Варіант 3. Factory Method + DI/ServiceLocator**
- Один Creator, який отримує всі продукти через DI.
- Вибір продукту — через DI.
- Плюси: легко підключати нові продукти.
- Мінуси: складніше відслідковувати залежності.

---

### Критика варіантів
- **Варіант 1** — класика GoF, ідеально для прозорої архітектури, легко розширювати, але багато класів.
- **Варіант 2** — компроміс між кількістю класів і SRP, але Creator стає "бог-об'єктом".
- **Варіант 3** — сучасний DI-friendly підхід, але ускладнює трасування та тестування.

---

### Висновок
**Вибрано: Варіант 1 (Класичний Factory Method, 1:1)**
- Максимальна ізоляція, чиста архітектура, кожен тип знижки — окремий клас.
- Контролер працює лише з DiscountCreator, не знає про продукти.
- Легко додавати нові типи знижок без зміни існуючого коду.
- Всі математичні розрахунки — у Product, логіка вибору — у Creator.

---

**Після підтвердження — реалізувати класичну Factory Method-структуру (Варіант 1, v3.0).**

## PROMPT v3.1.1: Знижки для всіх ролей та збереження суми зі знижкою (Minor Fix)

### Варіанти вирішення (ToT)

**Варіант 1. Універсальна логіка знижки для всіх користувачів**
- Застосовувати Factory Method для знижки незалежно від ролі (User/Admin/інше).
- Визначати тип знижки за тими ж правилами (перше замовлення, день народження) для будь-якого користувача, якщо він авторизований.
- Плюси: проста логіка, однаковий UX для всіх, адміністратор також отримує знижку.
- Мінуси: адміністратор може зловживати знижками (якщо це не бажано).

**Варіант 2. Додавати окрему перевірку ролі**
- Додавати додаткову перевірку: якщо користувач — адміністратор, застосовувати окрему знижку (або не застосовувати взагалі).
- Плюси: гнучкість, можна задати спеціальні правила для адміністраторів.
- Мінуси: ускладнення логіки, потрібно підтримувати окремі сценарії.

**Варіант 3. Винести логіку знижки у сервіс**
- Винести всю логіку вибору знижки у окремий DiscountService, який приймає User, Cart, Context.
- Сервіс повертає тип знижки, розмір, фінальну суму.
- Плюси: максимальна ізоляція, легко тестувати, можна розширювати правила.
- Мінуси: більше коду, але архітектурно чисто.

**Варіант 4. Зберігати суму зі знижкою у Order**
- Додати у Order поле DiscountedTotal.
- При підтвердженні замовлення зберігати саме фінальну суму (зі знижкою), а не лише повну.
- Плюси: коректна історія, правильна аналітика, відображення у адмінці.
- Мінуси: потрібно оновити модель Order та всі місця, де використовується TotalAmount.

---

### Критика
- Варіант 1 — простий, але не дає гнучкості для різних ролей.
- Варіант 2 — дозволяє кастомізувати політику для адміністраторів, але ускладнює код.
- Варіант 3 — найкращий для масштабування, але потребує рефакторингу.
- Варіант 4 — обов’язковий для коректного збереження історії замовлень.

---

### Висновок
**Вибрано: Варіант 1 + Варіант 4**
- Знижка застосовується для всіх авторизованих користувачів (включно з адміністратором).
- У Order додається поле DiscountedTotal (або FinalTotal), і при оформленні замовлення зберігається саме ця сума.
- Це забезпечує просту, прозору логіку та коректну історію замовлень.

---

**Після підтвердження — реалізувати збереження суми зі знижкою та універсальну логіку для всіх ролей (v3.1.1).**
